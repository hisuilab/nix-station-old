## ドキュメント

### 完全性
- [ ] 新しい開発者がリポジトリをセットアップするのに必要な手順が揃っているか
- [ ] 前提条件（必要ツール・バージョン）が明記されているか
- [ ] 手動で必要な作業（例: 秘密ファイルの作成）が明記されているか
- [ ] 各コマンドの実行環境（OS・対象ホスト）が明記されているか

### 正確性
- [ ] コメントや説明が現在の実装と一致しているか
- [ ] バージョン番号・パス・コマンドが最新か
- [ ] README に古い情報が残っていないか

### コメントの質
- [ ] WHY（なぜこう書いたか）が書かれているか
- [ ] WHAT（何をしているか）だけのコメントになっていないか
- [ ] 非自明な制約・回避策にコメントがあるか

## 設計

### 関心の分離
- [ ] 各層（例: hosts / modules / lib / flake）の責務が明確か
- [ ] ポリシーと仕組みが混在していないか
- [ ] 設定値が適切な層に置かれているか

### 拡張性
- [ ] 新しいホスト・ユーザー・モジュールを追加する手順が明確か
- [ ] 追加時に修正が必要な箇所が最小化されているか
- [ ] 同じリストや定義が複数箇所に分散していないか

### 一貫性
- [ ] 似た問題を同じパターンで解いているか
- [ ] 例外的な扱いが生じている箇所に理由があるか
- [ ] エラーハンドリングのパターンが統一されているか

### トレードオフの明示
- [ ] 意図的な制約（できないこと）がコメントまたはドキュメントに明記されているか
- [ ] 設計上の妥協点が将来の開発者に伝わるか

## テスト

### カバレッジ
- [ ] 新しく追加した機能にテストがあるか
- [ ] エッジケース（空入力・ファイル不在・権限エラーなど）がテストされているか
- [ ] 削除したコードが担保していた振る舞いが他でカバーされているか

### 独立性
- [ ] テストが実際の環境設定（実ホスト・実ユーザー）に依存していないか
- [ ] フィクスチャ・モックが適切に使われているか
- [ ] テストの実行順序に依存していないか

### CI との対応
- [ ] CI で実行されるチェックと flake の checks が対応しているか
- [ ] CI でスキップされているチェックがないか（特にクロスプラットフォーム）
- [ ] CI のジョブ名と flake の attribute 名が一致しているか

### 保守性
- [ ] ホスト・ツールを追加したときにテストが自動で追従するか
- [ ] テストのスタイルが統一されているか（例: `lib.runTests` vs 直接 `throw`）
- [ ] 失敗したときにどのテストが落ちたか特定しやすいか

## 実装

### 一貫性
- [ ] 同じ計算・パターンが複数箇所に重複していないか
- [ ] 変数のスコープ管理が統一されているか
- [ ] エラーメッセージのフォーマットが統一されているか

### 堅牢性
- [ ] 空文字・空ファイル・未設定変数のエッジケースを扱っているか
- [ ] 外部コマンド・ファイル操作の失敗を検出しているか
- [ ] 状態の不整合（書き込み成功・適用失敗など）が永続しないか

### 慣用表現（zsh）
- [ ] `[[ ]]` / `(( ))` を適切に使っているか
- [ ] `print` vs `echo` を適切に使い分けているか
- [ ] パラメータ展開・glob 修飾子を活用しているか
- [ ] `emulate -L zsh` などの環境保護が必要な箇所にあるか

### 慣用表現（Nix）
- [ ] `lib` 関数（`filterAttrs` / `optionalAttrs` / `mkMerge` など）を使える箇所で使っているか
- [ ] 不要な再計算・二重評価がないか
- [ ] `//` による上書きより `lib.mkMerge` が適切な箇所はないか
