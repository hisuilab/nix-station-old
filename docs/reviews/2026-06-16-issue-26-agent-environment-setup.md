# レビュー: エージェント開発環境の現状課題

> このレビューは [Claude Sonnet 4.6](https://claude.ai) によって 2026-06-16 に実施しました。
> 対象: nix-station における AI エージェント（Claude / Codex）を用いた開発ワークフローの現状課題

---

## レビュー観点

- エージェント精度 — セッション間の情報損失・誤操作・外部 API の変更追従
- 開発効率 — ワークフロー形式化・繰り返し指示の削減・マルチエージェント対応

## 指摘一覧

| # | 箇所 | 問題 | 対処案 |
|---|---|---|---|
| 1 | `CLAUDE.md`（未作成）| エージェントが毎セッション白紙状態で開始する。`.gitignore` で管理される「`guest.nix` / `test.nix` 以外のユーザープロファイルはコミット禁止」などクリティカルな制約、ブランチ命名規則（`type/issue-N-description`）、コミット粒度の指示がすべて口頭依存。前セッションでの git 破壊的操作（指摘 #5）もこれが一因 | `CLAUDE.md` を作成してセキュリティ制約・ワークフロー概要・命名規則・破壊的操作禁止事項を記載し、セッション開始時のコンテキストとして機能させる |
| 2 | `AGENTS.md`（未作成）| Codex など Claude 以外のエージェントがプロジェクト固有のルール（Nix flake 構成・ホスト追加手順・Brewfile 運用）を参照できない。複数エージェントを使い分ける場合に指示の一貫性が保てない | `AGENTS.md` を作成し、CLAUDE.md と同等の内容をエージェント非依存な形式で記載する |
| 3 | `.claude/commands/`（未作成）| レビュー作成 → 意思決定 → issue 作成 → ブランチ → 実装 → PR → マージ の各ステージを毎回口頭で指示している。ステージ間の手順が不定で、issue 作成より先に実装が始まることがある | `.claude/commands/review.md` / `decide.md` / `issue.md` / `implement.md` / `pr.md` を作成し、各ステージをスラッシュコマンドで起動できる状態にする |
| 4 | セッション引き継ぎ（コンテキスト圧縮）| コンテキストウィンドウ枯渇時のサマリーで意思決定の背景・実装詳細が失われる。前セッションでは「homebrew activation 順序問題」の全容が圧縮され、次セッションでの正確な再現が困難になった | セッション開始時の確認手順を `CLAUDE.md` に明記（「`docs/decisions/` の最新ファイルと `git log --oneline -10` を確認してから着手」）。重要な判断は必ず `docs/decisions/` に記録し再参照可能にする |
| 5 | 前セッション実績: git 破壊的操作 | `git reset --hard` 実行前にワーキングツリーの状態確認を省略し、ユーザーのローカル編集を消失させた。ユーザーから「今後気をつけてください」と指摘 | `CLAUDE.md` に「`git reset --hard` / `git checkout .` / `git clean -f` の前には必ず `git status` で未コミット変更を確認し、ユーザー承認を得る」を明記する |
| 6 | 前セッション実績: CI ハッシュ誤り | `docker/setup-qemu-action` のコミットハッシュを手書きで記載した際に 1 文字誤り（`...ff9c09cf...` vs 正 `...ff1c6c12...`）で CI が失敗した。エージェントがハッシュを推測・補完するリスクは根本的に除去できない | `CLAUDE.md` に「外部 GitHub Action のコミットハッシュは必ず `gh release view` または GitHub UI から取得し、推測で記入しない」を明記。CI 失敗時の調査フロー（`gh run view --log-failed`）も記載する |
| 7 | 前セッション実績: 廃止 API の使用 | nix-darwin で `system.activationScripts.postUserActivation` が廃止済みだったにもかかわらず使用し CI が失敗した。エージェントは学習時点の API を使用するため外部ライブラリの変更に気づきにくい | `CLAUDE.md` に「nix-darwin / home-manager の API を使う場合は nixpkgs リポジトリの対応モジュール定義を `grep` で確認してから実装する」を明記する |
| 8 | [`docs/reviews/template.md`](../../docs/reviews/template.md) / [`docs/decisions/template.md`](../../docs/decisions/template.md)| テンプレートは人間向けのフォーマット定義として書かれており、エージェントが「どのタイミングで・何を・どの順で」行うかの指示を含まない。レビュー観点の確認・指摘抽出・意思決定入力のいずれも都度口頭指示が必要 | `.claude/commands/` の各コマンドでテンプレートのどの項目をどの順に埋めるかを手順として記述し、エージェントがトリガーなしに従える形にする |

## 優先度まとめ

### 高（精度・安全性）

- **#1** `CLAUDE.md` 未作成: セッション間の制約引き継ぎ欠如・破壊的操作リスクの根本原因
- **#5** git 破壊的操作前の確認: 前セッションで実被害あり、ユーザー指摘済み

### 中（開発効率）

- **#3** `.claude/commands/` 未作成: ワークフロー各ステージが口頭指示依存
- **#4** セッション引き継ぎ: サマリー圧縮による情報損失・確認手順の不定
- **#8** テンプレートのエージェント対応: 手順書として機能していない

### 低（品質・再現性）

- **#2** `AGENTS.md` 未作成: Codex 対応が未整備
- **#6** CI ハッシュ誤り: 人間レビューで補完可能だが根本的な対策は限定的
- **#7** 廃止 API 使用: 実装前 `grep` 確認で検出可能
